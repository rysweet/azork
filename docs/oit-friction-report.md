## Friction found

### disc-intent — Resolve an ambiguous free-text intent

- **[confusing-message]** `make a new resource group please` — 'make a new resource group please' required disambiguation (did-you-mean)

## Live-tenant findings (surfaced and fixed)

- **Scalability:** on the live tenant (258 resource groups) AzZork's `az` backend issued one sequential `az resource list` per group, taking many minutes to build the world. Fixed by bounding rooms and resource enumeration (see AZORK_MAX_ROOMS).
- **Governance friction:** creating a Standard_LRS storage account was *denied by Azure Policy* ('Storage account public access should be disallowed'). Fixed by creating storage with `--allow-blob-public-access false`.
- **Creation dead-end:** on a fresh install, a natural request like 'create a storage account' resolved to nothing. Fixed by inferring the az domain and guiding the player to 'learn storage' (Resolution::LearnHint).

## Improvements fed back into AzZork

- Wired persistent graph memory into gameplay (rooms, resources, intents, friction) with new `recall`/`friction`/`memory` commands.
- Added intent-aware guidance so action words like `create`/`make`/`new` steer the player to the right `learn <group>` even before any capability is derived.
- Bounded the live `az` backend (AZORK_MAX_ROOMS / AZORK_MAX_RESOURCE_ROOMS) so AzZork is responsive on real subscriptions with hundreds of resource groups (this tenant has 258).
- Hardened the OIT storage-create path to satisfy tenant policy (public blob access disallowed) so live creation succeeds under governance controls.
- Auto-record unresolved intents as friction so gaps are captured for later fixing.
